# Reglas Permanentes del Proyecto - Angular 21

## 1. Estándar de Calidad
* Todo el código debe seguir un estándar de **Senior Frontend Engineer**.
* No se permiten soluciones rápidas que aumenten la deuda técnica.

## 2. Prohibiciones Críticas
* **Hardcodeos Prohibidos**: No se permiten textos de negocio, labels fijos, URLs, endpoints, keys, IDs o colores de negocio directamente en los componentes o plantillas.
* **No `any`**: El uso de `any` está prohibido en modelos principales y flujos críticos. Se requiere tipado fuerte obligatorio.
* **No `fetch`**: La estrategia de acceso a datos debe ser `HttpClient`.

## 3. Arquitectura y Tecnologías
* **Angular Moderno**: Priorizar `Standalone Components`, `inject()`, `Signals` (`computed`, `effect`) y Control Flow moderno (`@if`, `@for`).
* **HttpClient + Interceptores**: Centralizar la autenticación, headers y manejo de errores mediante interceptores.
* **Arquitectura Limpia**: Separación clara entre UI, lógica y capa de dominio/API.

## 4. Performance y Mantenibilidad
* Evitar lógica pesada en las plantillas.
* Optimizar renders y evitar cálculos repetitivos (usar `computed` para transformaciones de datos).
* Revisar y evaluar el impacto técnico de cada cambio antes de proceder.

## 5. Orden de Ejecución
* Antes de crear nuevas pantallas, se debe:
  1. Revisar y corregir lo que ya existe.
  2. Tipar modelos principales.
  3. Eliminar hardcodeos críticos.
  4. Sanear servicios y acceso a API.
* Recién después de cumplir estos puntos se procederá con nuevas funcionalidades.

## 6. Evaluación de Cambios (Checklist Senior)
Para cada modificación, se debe indicar explícitamente:
* Qué está bien y qué está mal.
* Cuál es el nivel de deuda técnica o riesgo.
* Qué hardcodeos se han eliminado.
* Por qué la solución es mantenible y escalable.
